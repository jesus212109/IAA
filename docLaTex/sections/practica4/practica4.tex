\chapter{Cuaderno de Laboratorio --- Práctica 4: Clasificación y el Valor del Error (Matriz de Costes)}

\section*{Objetivo}
Entrenar un clasificador probabilístico para un problema de detección de fraude bancario y estudiar cómo el umbral de decisión afecta a los errores cometidos y al coste total. El objetivo central no es únicamente medir el rendimiento predictivo, sino comprender el impacto económico real de las decisiones del modelo a partir de una \textbf{matriz de costes asimétrica}.

\section*{Descripción del problema}
Se trabaja con un problema de clasificación binaria sobre el dataset \texttt{fraude\_transacciones.csv}:
\begin{itemize}
    \item \textbf{Clase 0}: transacción normal.
    \item \textbf{Clase 1}: transacción fraudulenta.
\end{itemize}

Los errores tienen costes muy distintos:
\begin{itemize}
    \item \textbf{Falso Negativo (FN)}: no detectar un fraude $\rightarrow 200\,\text{€}$
    \item \textbf{Falso Positivo (FP)}: bloquear una transacción legítima $\rightarrow 10\,\text{€}$
\end{itemize}

La decisión final no debe evaluarse solo por el número de aciertos, sino por el \textbf{coste económico} asociado a los errores.

% =============================================================================
\section{Tarea 1 --- Análisis con el umbral por defecto (\texorpdfstring{$\tau = 0.5$}{tau=0.5})}
% =============================================================================

\subsection{Modelo utilizado}
Se ha entrenado una \textbf{Regresión Logística} (\texttt{LogisticRegression}, \texttt{max\_iter=1000}) sobre el 70\% del dataset (partición estratificada, \texttt{random\_state=42}) y evaluado sobre el 30\% restante.

\subsection{Matriz de confusión}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.52\textwidth]{matriz_confusion_umbral_0_5}
    \caption{Matriz de confusión para el umbral estándar $\tau = 0.5$.}
    \label{fig:mc_05}
\end{figure}

A partir de esta matriz se obtienen:
\begin{itemize}
    \item Verdaderos Positivos (TP): \textbf{16}
    \item Verdaderos Negativos (TN): \textbf{501}
    \item Falsos Positivos (FP): \textbf{1}
    \item Falsos Negativos (FN): \textbf{22}
\end{itemize}

\subsection{Cálculo del coste total}
\[
\text{Coste Total} = (FP \cdot 10) + (FN \cdot 200) = (1 \cdot 10) + (22 \cdot 200) = \mathbf{4410\,\text{€}}
\]

\subsection{Interpretación}
El umbral por defecto de 0.5 \textbf{no es razonable} en este contexto. Aunque el modelo comete muy pocos falsos positivos (solo 1), deja pasar 22 transacciones fraudulentas. Dado que el coste de un falso negativo (200\,€) es veinte veces el de un falso positivo (10\,€), el impacto económico resulta muy elevado. En un sistema de detección de fraude es fundamental minimizar los errores más caros; usar este umbral sin revisarlo sería un error de diseño grave.

\subsection{Análisis de distintos umbrales}
Se han extraído las probabilidades de la clase fraude para cada observación y se han evaluado umbrales $\tau \in [0, 1]$ en pasos de 0.05. La Tabla~\ref{tab:umbrales_coste} resume los resultados para los primeros tramos y el extremo superior.

\subsection{Observaciones}
Existe un claro compromiso (\textit{trade-off}) entre los dos tipos de error al variar $\tau$:
\begin{itemize}
    \item \textbf{Cuando el umbral baja} (hacia 0): se exige menos confianza para clasificar como fraude. Caen los Falsos Negativos (detectamos más fraudes), pero aumentan drásticamente los Falsos Positivos (bloqueamos más transacciones legítimas).
    \item \textbf{Cuando el umbral sube} (hacia 1): las alertas son mucho más selectivas. Los Falsos Positivos desaparecen, pero los Falsos Negativos se disparan (dejamos escapar la mayoría de los fraudes).
\end{itemize}
El umbral no es una constante sagrada: es una \textbf{decisión de diseño} que depende del contexto del problema y de la penalización relativa de cada tipo de error.

\begin{figure}[H]

    \label{fig:coste_umbral}

\subsection{Umbral óptimo}
\[
\tau^* = 0.05 \qquad \text{Coste mínimo} = 2920\,\text{€}
\]

\subsection{Interpretación}
El umbral óptimo ($\tau^* = 0.05$) es muy inferior a 0.5. Esto tiene todo el sentido dada la fuerte asimetría de la matriz de costes: el coste de un FN (200\,€) es 20 veces el de un FP (10\,€). Para minimizar el coste total conviene ser muy sensible a los fraudes, aunque eso suponga aceptar más falsos positivos. La estrategia más razonable para el banco es \textbf{conservadora}: es preferible molestar puntualmente a un cliente legítimo mediante un bloqueo temporal que permitir que un fraude se consume sin ser detectado.

\subsection{Curva Precision-Recall}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.62\textwidth]{curva_precision_recall}
    \caption{Curva Precision-Recall del clasificador. Average Precision $= 0.6551$.}
    \label{fig:pr}
\end{figure}

\subsection{Discusión}
En este problema con dataset desbalanceado (los fraudes son la clase minoritaria frente a un volumen masivo de transacciones normales), la \textbf{Curva Precision-Recall es más informativa} que la ROC.

La curva ROC puede resultar engañosa porque el False Positive Rate usa como denominador el inmenso total de negativos. Con tantos negativos, un aumento notable de FP apenas desplaza el FPR, manteniendo artificial­mente alto el AUC-ROC. La curva PR, en cambio, mide directamente la Precisión (TP / (TP + FP)), que cae rápidamente al aumentar los falsos positivos, reflejando con fidelidad cuán útil es el modelo para el departamento de seguridad a la hora de recuperar la clase minoritaria.

\section{Decisión final de ingeniería}
% =============================================================================

Recomiendo abandonar el umbral por defecto (0.5) y fijar $\tau = 0.05$ en producción. Con este umbral el coste esperado desciende de 4410\,€ a 2920\,€ sobre el conjunto de prueba. Si bien el número de bloqueos erróneos aumenta (de 1 a 112 FP), el coste unitario de un FP es veinte veces menor que el de un FN, por lo que la reducción de fraudes no detectados (de 22 a 9 FN) compensa con creces esa molestia operativa. El cliente legítimo bloqueado puede ser informado y desbloqueado en segundos a través de una alerta por SMS o app; el fraude consumado genera pérdidas directas e irrecuperables para la entidad.

Esta decisión se apoya en:
\begin{itemize}
    \item La matriz de costes asimétrica (FN 20$\times$ más caro que FP).
    \item La gráfica Coste vs.\ Umbral, que muestra el mínimo global en $\tau^* = 0.05$.
    \item La Curva Precision-Recall, más informativa para clases desbalanceadas.
    \item El contexto de negocio: la prioridad de un sistema antifraude es detectar fraudes, no minimizar alarmas.
\end{itemize}

% =============================================================================
\section{Conclusión final}
% =============================================================================

Esta práctica ha evidenciado que \textbf{en clasificación no todos los errores valen lo mismo}. Un modelo evaluado únicamente por su exactitud global puede parecer muy bueno mientras deja pasar sistemáticamente los errores más costosos.

Las conclusiones principales son:
\begin{enumerate}
    \item El umbral de decisión no es fijo ni universal: es un parámetro de diseño que debe elegirse en función del impacto económico de cada tipo de error.
    \item Bajar el umbral reduce FN a costa de más FP; subirlo hace lo contrario. Existe un único umbral que minimiza el coste total esperado.
    \item En datasets desbalanceados, la curva Precision-Recall refleja mejor la utilidad real del modelo que la curva ROC.
    \item Una evaluación rigurosa debe incorporar siempre el contexto y los costes del dominio, no solo métricas genéricas como la exactitud.
\end{enumerate}
